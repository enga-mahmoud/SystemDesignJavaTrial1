FILE: docker-compose.infra.yml
================================================================================
PURPOSE:
  Docker Compose file that starts ONLY the infrastructure dependencies (no
  application services). Use this first during development so you can run Spring
  Boot services locally via your IDE while Kafka, Redis, etc. run in Docker.

  Contains:
    - apache/kafka:3.7.1    Message broker for async event streaming (KRaft mode — no Zookeeper)
    - postgres-user:5432    PostgreSQL instance for user-service
    - postgres-product:5433 PostgreSQL instance for product-service
    - postgres-inventory:5434 PostgreSQL instance for inventory-service
    - postgres-order:5435   PostgreSQL instance for order-service
    - postgres-payment:5436 PostgreSQL instance for payment-service
    - redis:6379            In-memory store for cache, rate limiting, tokens
    - elasticsearch:9200    Search engine for product catalog
    - zipkin:9411           Distributed tracing collector UI

  NOTE: Uses apache/kafka:3.7.1 (official Apache image, not bitnami). Bitnami
  removed their images from Docker Hub in 2024. The official image uses bare
  Kafka property names as environment variables (no KAFKA_CFG_ prefix).
  KRaft mode eliminates Zookeeper — the broker and controller roles run in the
  same single container (KAFKA_PROCESS_ROLES=broker,controller).

  Each PostgreSQL instance runs on a different HOST PORT but the same container
  port (5432). This simulates separate databases while running on one machine.
  In production, each service would have its own PostgreSQL cluster.

FILE: infrastructure/init-scripts/01-user-db.sql
================================================================================
PURPOSE:
  PostgreSQL initialization script for user-service's database. Executed once
  when the postgres-user container first starts. Creates the schema that
  user-service uses. In production, Flyway or Liquibase would manage migrations
  instead of init scripts.

SQL TABLES:

  TABLE: users
    id            UUID PRIMARY KEY  - Globally unique user identifier.
                                      gen_random_uuid() uses pgcrypto extension.
                                      UUID is better than SERIAL here because:
                                        1. No central sequence, safe to generate in app
                                        2. Does not expose row count to clients
    email         VARCHAR(255)      - User's login email. UNIQUE constraint ensures
                                      no two users share the same email. NOT NULL
                                      prevents empty registrations.
    password_hash VARCHAR(255)      - BCrypt output (always 60 chars but using 255
                                      for future algorithm changes). NEVER stores
                                      plaintext password.
    role          VARCHAR(50)       - Access level: USER (default), ADMIN, VENDOR.
                                      VARCHAR not ENUM because adding new roles in
                                      PostgreSQL ENUM requires ALTER TYPE which locks
                                      the table.
    created_at    TIMESTAMPTZ       - Timestamp WITH timezone. Always store in UTC.
                                      DEFAULT NOW() sets automatically on INSERT.
    updated_at    TIMESTAMPTZ       - Updated by application before every save.
                                      In production, use a trigger to auto-update.

  INDEX: idx_users_email ON users(email)
    Enables fast lookup by email on every login request. Without this index,
    every login would require a full table scan — O(n) instead of O(log n).

FILE: infrastructure/init-scripts/02-product-db.sql
================================================================================
PURPOSE:
  Initializes the product-service database schema. Three tables: categories,
  products, and product_outbox (for the Outbox Pattern).

SQL TABLES:

  TABLE: categories
    id        UUID PRIMARY KEY    - Category unique ID
    name      VARCHAR(255)        - Display name (e.g., "Electronics", "Laptops")
    parent_id UUID REFERENCES     - Self-referencing FK for hierarchical categories.
                                    parent_id IS NULL = root category.
                                    parent_id IS NOT NULL = subcategory.
                                    Example: Electronics → Laptops → Gaming Laptops

  TABLE: products
    id          UUID PRIMARY KEY  - Product unique ID
    name        VARCHAR(500)      - Product title. 500 chars allows long descriptive names.
    description TEXT              - Full product description. TEXT has no length limit.
    price       NUMERIC(12,2)     - Monetary amount. NEVER use FLOAT for money
                                    (floating point imprecision). NUMERIC(12,2) stores
                                    up to $9,999,999,999.99 with exact precision.
    category_id UUID REFERENCES   - Foreign key to categories. NULL allowed (uncategorized).
    vendor_id   UUID              - ID of the vendor who owns this product. NOT a FK
                                    because vendor data is in user-service (different DB).
                                    Cross-service FKs do not exist in microservices.
    status      VARCHAR(50)       - ACTIVE (visible), DELETED (soft-deleted), OUT_OF_STOCK.
                                    Soft delete: set status=DELETED instead of DELETE row.
                                    Preserves referential integrity for order history.
    created_at  TIMESTAMPTZ       - Immutable creation timestamp
    updated_at  TIMESTAMPTZ       - Last modification timestamp

  TABLE: product_outbox
    id           UUID PRIMARY KEY - Outbox record ID. UUID generated by Hibernate
                                    (@GeneratedValue with UUID strategy) before INSERT,
                                    so the application knows the ID without a DB round-trip.
    aggregate_id UUID             - The product ID this event relates to. Called "aggregate_id"
                                    (not "product_id") because Outbox Pattern is domain-agnostic:
                                    any aggregate (product, order, payment) can have an outbox.
                                    Mapped from Product.getId() at write time.
    event_type   VARCHAR(100)     - Kafka topic name: "product.created", "product.updated",
                                    "product.deleted". The OutboxPublisher uses this as the
                                    Kafka topic string directly — no enum-to-string conversion
                                    needed at publish time. VARCHAR(100) is generous; current
                                    values are all under 20 chars.
    payload      JSONB            - Full event payload as JSON. JSONB (Binary JSON) vs TEXT:
                                      • Parsed and stored in a binary format internally.
                                      • Supports GIN indexes for fast key/value lookups.
                                      • Validates JSON on INSERT — malformed JSON is rejected.
                                      • Queryable: SELECT payload->>'name' FROM product_outbox.
                                    The Java entity uses @JdbcTypeCode(SqlTypes.JSON) so
                                    Hibernate 6 binds the String as the JSON JDBC type, not
                                    VARCHAR. Without this annotation, PostgreSQL rejects the
                                    INSERT with: "column payload is of type jsonb but expression
                                    is of type character varying".
                                    columnDefinition="jsonb" on the entity controls DDL only
                                    (the CREATE TABLE statement) and does NOT affect the JDBC
                                    bind type — both annotations are required together.
    created_at   TIMESTAMPTZ      - When this outbox row was written. Set at entity construction
                                    time (LocalDateTime.now()) — NOT via JPA @CreatedDate, to
                                    keep the outbox entity self-contained. @Builder.Default is
                                    required on the field so Lombok's @Builder does not silently
                                    null it out (Lombok @Builder ignores field initializers
                                    without @Builder.Default).
    published    BOOLEAN          - FALSE on INSERT. Set to TRUE by OutboxPublisher only after
                                    kafkaTemplate.send() succeeds AND the transaction commits.
                                    If Kafka is down, published stays FALSE permanently for that
                                    row — the scheduler retries it on the next poll cycle.
                                    The default value is defined both at the DB level (DEFAULT false
                                    in SQL) and in the Java entity (= false field initializer)
                                    so both the raw SQL path and the JPA path are consistent.

  LIFECYCLE OF A PRODUCT_OUTBOX ROW:
    1. ProductService.createProduct() / updateProduct() / deleteProduct() opens a transaction.
    2. Product row is written to the products table.
    3. ProductOutbox entity is built with published=false, eventType=<topic>, payload=<JSON>.
    4. outboxRepository.save(outbox) → INSERT INTO product_outbox (..., published=false).
    5. Transaction commits — both the product row and the outbox row are durable together.
       This is the atomicity guarantee: either both writes land, or neither does.
    6. OutboxPublisher @Scheduled(fixedDelay=1000) wakes up, queries unpublished rows:
         SELECT * FROM product_outbox WHERE published=false ORDER BY created_at
    7. For each row, kafkaTemplate.send(eventType, aggregateId, payload) is called.
    8. On success, row.setPublished(true). outboxRepository.saveAll(rows) → batch UPDATE.
    9. The row stays in the table forever (published=true). Cleanup is a separate concern
       (a periodic DELETE job or table partitioning) not yet implemented in this project.

  WHAT HAPPENS WHEN KAFKA IS DOWN:
    • kafkaTemplate.send() throws or times out inside OutboxPublisher.
    • The @Transactional on OutboxPublisher.publishPending() does NOT commit published=true.
    • The row stays published=false.
    • The scheduler retries the same rows on the next 1-second cycle.
    • The product_outbox table accumulates unpublished rows during the outage.
    • When Kafka recovers, all accumulated rows drain automatically in order (ORDER BY created_at).
    • No manual intervention required — at-least-once delivery is guaranteed.

  INDEX: idx_outbox_unpublished ON product_outbox(published, created_at) WHERE published = false
    Partial index. Only indexes rows where published=false. The OutboxPublisher queries
    "WHERE published=false ORDER BY created_at". This partial index makes that query
    O(log n) on the unpublished set only, not the entire table.
    WHY PARTIAL (not a regular index on published):
      • A regular index on (published, created_at) would index ALL rows, including the
        ever-growing set of published=true rows — wasting space and slowing down writes.
      • A partial index on WHERE published=false stays tiny: it only tracks the rows
        the scheduler actually needs to read. Once a row is marked published=true,
        PostgreSQL automatically removes it from this index.
      • This is the canonical pattern for polling-based outbox tables: the hot set
        (unpublished) is small, the cold set (published) is large and irrelevant to the query.

FILE: infrastructure/init-scripts/04-order-db.sql
================================================================================
PURPOSE:
  Initializes order-service's three-table schema supporting Event Sourcing,
  CQRS read model, and Saga state persistence. These three tables are tightly
  coupled by design — each one serves a distinct architectural pattern, and
  together they enable the full order lifecycle:
    order_events  → Event Sourcing (write side, append-only truth)
    orders_read_model → CQRS (pre-built projection for fast reads)
    order_saga    → Saga Orchestration (distributed transaction state machine)

SQL TABLES:

  TABLE: order_events  (Event Store)
  ─────────────────────────────────
  This is the system of record for all order data. No order state is stored
  directly — instead, what happened is stored as a sequence of immutable events.
  The current state of any order is derived by replaying these events in order.

    id          UUID PRIMARY KEY  - Unique ID for this event record. Generated by
                                    Hibernate (@GeneratedValue UUID) before INSERT —
                                    no round-trip to the DB needed.
    order_id    UUID              - Which order this event belongs to. Not a FK —
                                    there is no "orders" table in Event Sourcing.
                                    The order's identity IS the stream of events
                                    keyed by this UUID.
    event_type  VARCHAR(100)      - String discriminator: "OrderPlacedEvent",
                                    "OrderConfirmedEvent", "OrderCancelledEvent".
                                    Used by OrderProjector.applyEvent() to switch
                                    on the correct projection logic. VARCHAR(100)
                                    is generous; all current values are under 30 chars.
    payload     JSONB             - Complete event data serialized as JSON. JSONB
                                    (Binary JSON) is used over TEXT/VARCHAR because:
                                      • Validated on INSERT — malformed JSON rejected.
                                      • Supports key/value queries: payload->>'orderId'.
                                      • Stored in binary, faster to read than TEXT.
                                    The Java entity uses @JdbcTypeCode(SqlTypes.JSON)
                                    so Hibernate 6 binds the Java String as the JSON
                                    wire type, not VARCHAR. Without this, PostgreSQL
                                    rejects the INSERT with:
                                      "column payload is of type jsonb but expression
                                       is of type character varying"
                                    columnDefinition="jsonb" on the entity is a DDL
                                    hint only (controls the CREATE TABLE statement)
                                    and does NOT affect JDBC binding — both annotations
                                    are always required together.
    sequence_no INT               - 1-based position of this event in the order's
                                    event stream. The first event (OrderPlacedEvent)
                                    is always sequenceNo=1. Subsequent events increment.
                                    Used in:
                                      • Replay: ORDER BY sequence_no ASC ensures events
                                        are applied in the correct causal order.
                                      • Time-travel: replay up to sequenceNo=N to see
                                        past state at a specific point in history.
                                      • Concurrency detection: the UNIQUE constraint
                                        (see below) rejects duplicate writes at the same
                                        position.
    occurred_at TIMESTAMPTZ       - Business time: when the domain event actually
                                    occurred. Distinct from the DB INSERT timestamp.
                                    Used for time-travel debugging (replay events up
                                    to a point in time). Set at entity construction
                                    via @Builder.Default = LocalDateTime.now(). Without
                                    @Builder.Default, Lombok @Builder ignores the field
                                    initializer and occurred_at would be null on INSERT,
                                    violating the implicit NOT NULL expectation.

    UNIQUE (order_id, sequence_no)
      Prevents two events from occupying the same position in an order's stream.
      This is the optimistic concurrency guard for the event store:
        • If sequence_no=3 exists for order X, a second INSERT with sequence_no=3
          for the same order will fail with a unique constraint violation.
        • This protects against race conditions: two concurrent command handlers
          both computing "next sequenceNo = 3" and trying to write simultaneously.
          One succeeds; the other gets a constraint violation and must retry.
      Append-only guarantee: events are never updated or deleted after INSERT.
      The table is a permanent, immutable audit log.

    HIBERNATE BATCH SETTINGS (from application.yml):
      hibernate.jdbc.batch_size=50 and hibernate.order_inserts=true allow
      Hibernate to batch multiple OrderEventEntity INSERTs into a single JDBC
      call, reducing DB round-trips when multiple events are saved together.

  TABLE: orders_read_model  (CQRS Read Side)
  ───────────────────────────────────────────
  A pre-built, denormalized snapshot of the current order state. It is never
  the source of truth — it is derived from order_events and can be fully
  rebuilt from scratch by replaying all events. It exists solely to serve
  fast, JOIN-free reads. Without it, every GET /orders/{id} would have to
  replay the entire event stream — expensive and slow.

    id          UUID PRIMARY KEY  - Matches the order ID (same UUID used in
                                    order_events.order_id). There is no separate
                                    auto-increment ID — the order UUID is the key
                                    on both the event side and the read side.
    user_id     UUID              - Who placed the order. Denormalized from the
                                    OrderPlacedEvent payload at projection time.
                                    Stored here so queries like "all orders for
                                    user X" don't need to parse JSONB.
    status      VARCHAR(50)       - Current order status: PENDING, CONFIRMED,
                                    CANCELLED. Projected from the latest event:
                                      OrderPlacedEvent     → PENDING
                                      OrderConfirmedEvent  → CONFIRMED
                                      OrderCancelledEvent  → CANCELLED
                                    Updated by OrderProjector on every state-changing
                                    event. Reading status from here is O(1); reading
                                    it from the event store would be O(n events).
    total       NUMERIC(12,2)     - Order total in the same NUMERIC(12,2) format as
                                    product prices (exact decimal, no floating-point
                                    drift). Denormalized from OrderPlacedEvent.total
                                    at projection time, so display doesn't require
                                    recomputing sum(quantity * unitPrice).
    items_json  JSONB             - Full array of order items. Denormalized so reads
                                    return the complete order in one SELECT with no JOINs.
                                    Acceptable here because:
                                      a) This table can be reconstructed from events
                                         at any time — it is not the source of truth.
                                      b) Order items don't change after placement —
                                         denormalization doesn't cause update anomalies.
                                    Same @JdbcTypeCode(SqlTypes.JSON) requirement as
                                    order_events.payload for Hibernate 6 JSONB binding.
    created_at  TIMESTAMPTZ       - Order creation time (from OrderPlacedEvent.occurredAt).
                                    Immutable after the projection is first written.
    updated_at  TIMESTAMPTZ       - When the projection was last updated. Advances with
                                    each new event (OrderConfirmedEvent, OrderCancelledEvent).
                                    Set by @Builder.Default = LocalDateTime.now() in the
                                    entity and manually updated by OrderProjector on
                                    each state transition.

    HOW THE READ MODEL IS MAINTAINED:
      OrderProjector is called synchronously within the PlaceOrderCommandHandler
      transaction. Each time an event is written to order_events, the projector
      immediately updates orders_read_model in the same DB transaction. This means
      reads are always consistent with the latest command — no eventual consistency
      lag for queries in this implementation.

    HOW THE READ MODEL IS QUERIED:
      GetOrderQueryHandler reads ONLY from orders_read_model — it never touches
      order_events. This is the CQRS separation: the query handler has no knowledge
      of the event store. A corrupted or missing read model can be rebuilt by
      EventReplayService replaying all events without any data loss.

  TABLE: order_saga  (Saga State Machine)
  ────────────────────────────────────────
  Tracks the multi-step distributed transaction coordinating order placement
  across order-service, inventory-service, and payment-service. One row per order.
  Each state transition is a DB UPDATE, making the saga durable across crashes.

    id          UUID PRIMARY KEY  - Unique saga record ID. One saga per order, but
                                    the saga gets its own UUID separate from order_id
                                    to keep it extensible (a future retry saga could
                                    be a new row with a new id for the same order_id).
    order_id    UUID UNIQUE       - The order this saga coordinates. UNIQUE enforces
                                    the invariant "at most one active saga per order"
                                    at the DB level. A second saga for the same order
                                    fails with a unique constraint violation — preventing
                                    duplicate saga execution from race conditions or
                                    duplicate Kafka delivery.
    state       VARCHAR(50)       - Current position in the saga state machine:
                                      PENDING              → Saga started, awaiting
                                                             inventory reservation result.
                                      INVENTORY_RESERVED   → Inventory reserved successfully.
                                                             Awaiting payment result.
                                      PAYMENT_PROCESSING   → Payment initiated (if explicitly
                                                             tracked; payment-service may
                                                             consume order.placed directly).
                                      CONFIRMED            → Both inventory and payment
                                                             succeeded. Order fulfilled.
                                      CANCELLED            → Saga compensated. Either
                                                             inventory failed or payment
                                                             failed and inventory was released.
                                    State transitions are validated in OrderSaga before
                                    each UPDATE — processing an event in the wrong state
                                    is silently ignored (idempotency guard).
    saga_data   JSONB             - Arbitrary JSON context carried across saga steps.
                                    Starts as "{}" on saga creation. Populated as the
                                    saga progresses. Currently stores reservation data:
                                      {"reservations": [{"skuId": "...", "quantity": 3}]}
                                    WHY STORE THIS IN THE DB?
                                      If order-service crashes between inventory reservation
                                      and payment failure handling, saga_data is durably
                                      stored in PostgreSQL. On restart, the saga reads this
                                      column and knows exactly which reservations to release
                                      — no data is lost. Without this, a crash between steps
                                      would leave inventory permanently reserved with no way
                                      to compensate.
                                    The Java type for deserialization is OrderSagaState,
                                    which holds List<Reservation> (skuId + quantity pairs).
                                    Same @JdbcTypeCode(SqlTypes.JSON) requirement as all
                                    other JSONB columns in this service.
    created_at  TIMESTAMPTZ       - When the saga started (when the order was placed).
                                    Immutable after creation. Set via @Builder.Default.
    updated_at  TIMESTAMPTZ       - When the saga last transitioned state. Updated
                                    manually by OrderSaga.onInventoryReserved(),
                                    onPaymentFailed(), etc. on every state change.
                                    Both timestamps require @Builder.Default in the
                                    entity — same reason as all other timestamp fields:
                                    Lombok @Builder ignores field initializers without it.

  SAGA STATE MACHINE FLOW:
    1. PlaceOrderCommandHandler creates order_saga row with state=PENDING in the
       same transaction as the order_events INSERT. Both writes are atomic.
    2. order.placed event is published to Kafka.
    3. inventory-service consumes order.placed → attempts stock reservation:
         SUCCESS → publishes inventory.reserved
         FAILURE → publishes inventory.failed
    4. OrderSaga.onInventoryReserved():
         Precondition: state == PENDING (ignore if already processed — idempotency)
         Stores {reservations} in saga_data (for compensation if needed later).
         Transitions: PENDING → INVENTORY_RESERVED
    5. payment-service independently consumes order.placed → charges customer:
         SUCCESS → publishes payment.completed
         FAILURE → publishes payment.failed
    6a. OrderSaga.onPaymentCompleted():
         Transitions: INVENTORY_RESERVED → CONFIRMED
         Publishes order.confirmed → notification-service sends confirmation email.
    6b. OrderSaga.onPaymentFailed():
         Reads saga_data to extract which reservations to release.
         Publishes inventory.release-request for each reservation.
         Transitions: INVENTORY_RESERVED → PAYMENT_FAILED_COMPENSATING → CANCELLED
         Publishes order.cancelled → notification-service sends cancellation email.

  RELATIONSHIP BETWEEN ALL THREE TABLES:
    A single order placement writes to all three tables in one transaction:
      INSERT order_events (sequenceNo=1, OrderPlacedEvent)
      INSERT orders_read_model (status=PENDING)
      INSERT order_saga (state=PENDING)
    Every subsequent saga step writes to:
      order_events (new event row, sequenceNo increments)
      orders_read_model (UPDATE status)
      order_saga (UPDATE state, saga_data)
    The event store is the authoritative log. The read model and saga table
    are derived state that can be rebuilt from order_events if corrupted.

PURPOSE:
  Maven build file for the API Gateway service. Declares dependencies and build
  configuration. The parent POM inherits Spring Boot defaults.

KEY DEPENDENCIES:
  spring-boot-starter-parent      - Sets Spring Boot version (3.x). Manages all
                                    dependency versions (no need to specify versions
                                    for Spring libraries explicitly).

  spring-cloud-dependencies (BOM) - Spring Cloud version alignment. Imported as
                                    dependencyManagement to ensure all Spring Cloud
                                    modules are compatible versions.

  spring-cloud-starter-gateway    - Spring Cloud Gateway. Provides reactive (Netty)
                                    HTTP server with route matching, filters, and
                                    load balancing. Uses WebFlux not Servlet/Tomcat.

  spring-cloud-starter-netflix-eureka-client
                                  - Registers this gateway with Eureka for service
                                    discovery. Enables `lb://service-name` routing
                                    where lb = load-balanced DNS resolution via Eureka.

  spring-cloud-starter-config     - Fetches application.yml from Config Server on
                                    startup instead of using local application.yml.

  spring-boot-starter-security     - Spring Security. Provides SecurityWebFilterChain
                                    (reactive variant from spring-security-webflux).
                                    Required for the SecurityConfig bean that disables
                                    CSRF and sets the gateway to permit all requests
                                    (auth is enforced by JwtAuthFilter, not Spring Security).

  spring-boot-starter-data-redis-reactive
                                  - Reactive Redis client (Lettuce). Used for the
                                    rate limiting Lua script execution. Must be reactive
                                    because the gateway is reactive (WebFlux).

  io.jsonwebtoken:jjwt-api        - JJWT library for parsing and validating JWTs.
  io.jsonwebtoken:jjwt-impl       - JJWT implementation (loaded at runtime).
  io.jsonwebtoken:jjwt-jackson    - JJWT Jackson integration for JSON serialization.

  resilience4j-spring-boot2       - Circuit breaker, rate limiter, retry patterns.
                                    Integrates with Spring Cloud Gateway.

  micrometer-tracing-bridge-brave - Distributed tracing. Instruments WebFlux to
                                    propagate B3 trace headers (TraceId, SpanId) to
                                    downstream services and to Zipkin.

  spring-boot-starter-actuator    - Exposes /actuator/health, /actuator/prometheus
                                    endpoints for health checks and metrics.

FILE: api-gateway/src/main/java/com/ecommerce/gateway/GatewayApplication.java

PURPOSE:
  Entry point for the API Gateway Spring Boot application. The @SpringBootApplication
  annotation triggers component scanning, auto-configuration, and the Spring
  context initialization.

CLASS: GatewayApplication

  ANNOTATION: @SpringBootApplication
    Shorthand for three annotations combined:
      @Configuration       - This class can define @Bean methods
      @EnableAutoConfiguration - Spring Boot scans classpath and auto-configures
                               (e.g., detects spring-cloud-gateway on classpath,
                               auto-configures RouteLocator)
      @ComponentScan       - Scans com.ecommerce.gateway.** for @Component,
                             @Service, @Controller, etc.

  METHOD: main(String[] args)
    Parameters:  args - command-line arguments (e.g., --server.port=8080)
    Returns:     void
    Description: SpringApplication.run() bootstraps the Spring context.
                 For Spring Cloud Gateway this starts a Netty HTTP server (not Tomcat)
                 because the gateway is built on WebFlux (reactive/non-blocking).
                 Netty can handle thousands of concurrent connections with far fewer
                 threads than a thread-per-request Tomcat server.

FILE: api-gateway/src/main/java/com/ecommerce/gateway/config/GatewayConfig.java
================================================================================
PURPOSE:
  Defines all route rules for the API Gateway. Each route maps a URL path prefix
  to a downstream service (resolved via Eureka load balancing). Also configures
  Resilience4j circuit breakers per route.

CLASS: GatewayConfig

  ANNOTATION: @Configuration
    Marks this as a configuration class. Spring will invoke all @Bean methods
    and register their return values in the application context.

  METHOD: routeLocator(RouteLocatorBuilder builder)
    Returns: RouteLocator - the complete set of routing rules
    Description:
      Uses the fluent RouteLocatorBuilder DSL to define routes.
      Each route has:
        id:        human-readable name for logging/metrics
        predicate: condition to match (path pattern, method, header, etc.)
        filters:   transformations to apply before forwarding
        uri:       destination (lb:// = load-balanced via Eureka)

    ROUTE 1: user-service route
      Path: /api/auth/**
      URI:  lb://user-service
      No auth filter (login and register must be unauthenticated).
      StripPrefix=1 strips the /api prefix before forwarding to user-service.

    ROUTE 2: product-service route
      Path: /api/products/**
      URI:  lb://product-service
      CircuitBreaker filter: if product-service fails 5x in a row,
        circuit opens and requests go to /fallback/products instead.

    ROUTE 3: search-service route
      Path: /api/search/**
      URI:  lb://search-service
      Public (no JWT required for GET). The JwtAuthFilter skips this path.

    ROUTE 4: order-service route
      Path: /api/orders/**
      URI:  lb://order-service
      Requires authentication (JwtAuthFilter validates token).

    ROUTE 5: inventory-service route
      Path: /api/inventory/**
      URI:  lb://inventory-service

FILE: api-gateway/src/main/java/com/ecommerce/gateway/filter/JwtAuthFilter.java
================================================================================
PURPOSE:
  A Spring Cloud Gateway GlobalFilter that runs on every incoming request.
  Validates the JWT token in the Authorization header using the RSA public key.
  On success, injects the user's ID and role as request headers so downstream
  services can identify the caller without re-validating the token themselves.
  On failure or missing token, returns HTTP 401 (unless path is whitelisted).

CLASS: JwtAuthFilter implements GlobalFilter, Ordered

  INTERFACE GlobalFilter
    A filter that applies to ALL routes (not per-route). Every request passes
    through this filter regardless of which route it matches.

  INTERFACE Ordered
    getOrder() returns -1, which means this filter runs BEFORE the routing
    filter (which forwards the request downstream). Security check happens
    first.

  FIELD: JwtUtil jwtUtil
    Injected utility bean that holds the RSA public key and performs token
    parsing. Defined as a bean in JwtConfig.java.

  FIELD: List<String> PUBLIC_PATHS
    Paths that skip JWT validation for ALL methods: /api/auth/, /api/search, /actuator/
    Additional paths that skip validation for GET only: /api/products

    IMPORTANT — no trailing slash on /api/search:
      The check uses path.startsWith(publicPath). The request URL for a search is
      /api/search?q=... — the path is /api/search (no slash before ?). If the entry
      were /api/search/ it would NOT match, and every search would return 401.
      /api/auth/ keeps its trailing slash because auth paths are /api/auth/login,
      /api/auth/register etc., which always have a subpath after the slash.

  METHOD: filter(ServerWebExchange exchange, GatewayFilterChain chain)
    Parameters:
      exchange - The current reactive HTTP request/response context.
                 ServerWebExchange is the WebFlux equivalent of HttpServletRequest.
                 Unlike Servlet, it wraps BOTH request and response in one object.
      chain    - The remaining filter pipeline. Calling chain.filter(exchange)
                 passes control to the next filter (and eventually to routing).
    Returns: Mono<Void>
      Reactive return type. Mono<Void> means "do something and complete" without
      returning a value. The gateway framework subscribes to this Mono.
    Logic:
      1. Extract the request path from exchange.getRequest().getPath().value()
      2. If path matches any PUBLIC_PATHS pattern, call chain.filter(exchange)
         immediately (skip auth)
      3. Extract Authorization header. If null or doesn't start with "Bearer ",
         call unauthorizedResponse(exchange) which writes 401 JSON.
      4. Extract the token string by stripping "Bearer " prefix (7 characters).
      5. Call jwtUtil.validateAndParse(token):
           - Parses the JWT using the RSA public key
           - Verifies signature, expiry, and issuer claim
           - Returns Claims object (map of all JWT claims)
      6. Extract userId and role from Claims.
      7. Mutate the downstream request:
           ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
             .header("X-User-Id", userId)
             .header("X-User-Role", role)
             .build();
         The mutated request is what downstream services receive.
      8. Return chain.filter(exchange.mutate().request(mutatedRequest).build())
      9. On JwtException (expired, malformed, invalid signature):
           Call unauthorizedResponse(exchange) — downstream never sees the request.

  METHOD: unauthorizedResponse(ServerWebExchange exchange)
    Parameters: exchange - current request/response context
    Returns:    Mono<Void>
    Description:
      Sets response status to 401 UNAUTHORIZED.
      Sets Content-Type: application/json.
      Writes JSON body: {"error":"Unauthorized","message":"Invalid or missing token"}
      Uses DataBufferUtils with exchange.getResponse().writeWith() for reactive writing.

  METHOD: getOrder()
    Returns: int (-1)
    Why -1? Spring Cloud Gateway's RoutePredicateHandlerMapping has order 1.
    By returning -1, this filter runs BEFORE route matching, ensuring unauthenticated
    requests are rejected before the gateway even resolves the downstream service.

FILE: api-gateway/src/main/java/com/ecommerce/gateway/filter/RateLimitFilter.java
================================================================================
PURPOSE:
  Implements sliding window rate limiting using Redis. Allows 100 requests per
  60 seconds per client IP. Uses a Lua script executed atomically in Redis to
  avoid race conditions between the check and the increment operations.

CLASS: RateLimitFilter implements GlobalFilter, Ordered

  FIELD: ReactiveRedisTemplate<String, String> redisTemplate
    Reactive (non-blocking) Redis client. Must be reactive because the gateway
    runs on WebFlux. A blocking Redis call would block the Netty event loop thread,
    degrading performance dramatically.

  FIELD: int MAX_REQUESTS = 100
    Maximum allowed requests per window. Configurable via @Value("${rate.limit.max}").

  FIELD: int WINDOW_SECONDS = 60
    Time window in seconds. Redis key expires after this many seconds.

  METHOD: filter(ServerWebExchange exchange, GatewayFilterChain chain)
    Logic:
      1. Extract client IP from exchange.getRequest().getRemoteAddress().getAddress().getHostAddress()
         In production behind a proxy, use X-Forwarded-For header instead.
      2. Build Redis key: "rate:" + clientIp
      3. Execute Lua script atomically:
           local count = redis.call('INCR', KEYS[1])
           if count == 1 then
             redis.call('EXPIRE', KEYS[1], ARGV[1])
           end
           return count
         INCR creates the key if it doesn't exist (count=1) and increments it.
         EXPIRE is set only on the first request (count==1) to start the 60-second window.
         Atomic execution: both INCR and EXPIRE happen together, no race condition.
      4. If count <= MAX_REQUESTS: chain.filter(exchange) (allow through)
      5. If count > MAX_REQUESTS:
           Set response status 429 Too Many Requests
           Set header Retry-After: 60
           Write JSON body: {"error":"Rate limit exceeded","retryAfter":60}

  METHOD: getOrder()
    Returns: -2
    Runs before JwtAuthFilter (order -1). Rate limiting before auth prevents
    attackers from burning auth processing capacity with high-volume requests.

FILE: api-gateway/src/main/java/com/ecommerce/gateway/filter/CorrelationIdFilter.java
================================================================================
PURPOSE:
  Injects a unique X-Correlation-Id header into every request. Downstream services
  log this ID. Allows tracing a single user request across all service logs by
  searching for the correlation ID.

CLASS: CorrelationIdFilter implements GlobalFilter, Ordered

  FIELD: String CORRELATION_ID_HEADER = "X-Correlation-Id"

  METHOD: filter(ServerWebExchange exchange, GatewayFilterChain chain)
    Logic:
      1. Check if X-Correlation-Id already present (client may send their own).
         If present, use it (allows client-side request tracking).
         If absent, generate UUID.randomUUID().toString()
      2. Add X-Correlation-Id to the downstream REQUEST via exchange.mutate().
      3. Register a beforeCommit() hook on the response so the correlation ID
         is also added to the RESPONSE headers just before they are written to
         the wire (while they are still mutable):
           mutatedExchange.getResponse().beforeCommit(() -> {
               mutatedExchange.getResponse().getHeaders()
                   .add(CORRELATION_ID_HEADER, finalCorrelationId);
               return Mono.empty();
           });
      4. Return chain.filter(mutatedExchange).

    WHY beforeCommit() AND NOT .then(Mono.fromRunnable(...))?
      The original implementation used:
        chain.filter(...).then(Mono.fromRunnable(() ->
            exchange.getResponse().getHeaders().add(...)))
      This ran AFTER chain.filter() completed. By that time the response body
      is already fully written and the headers object is frozen as a
      ReadOnlyHttpHeaders. Calling .add() on ReadOnlyHttpHeaders throws
      UnsupportedOperationException. Reactor Netty catches this, logs
      "Error finishing response. Closing connection", and closes the TCP
      connection without writing the final 0\r\n\r\n chunk. Every proxied
      request received ERR_INCOMPLETE_CHUNKED_ENCODING in the browser.
      beforeCommit() runs BEFORE the headers are sent, while they are still
      writable — this is Spring WebFlux's intended API for filter-level
      response header mutation.

  METHOD: getOrder()
    Returns: -3 (runs first, before rate limiting and auth)
    All subsequent filters and service calls have the correlation ID available.

FILE: api-gateway/src/main/java/com/ecommerce/gateway/util/JwtUtil.java
================================================================================
PURPOSE:
  Utility class for JWT validation in the API Gateway. Holds the RSA public key
  loaded from Config Server and provides a single validateAndParse() method.
  The gateway only VERIFIES tokens — it never CREATES them (that's user-service's job).

CLASS: JwtUtil

  ANNOTATION: @Component
    Registers this class as a Spring bean, injectable anywhere via @Autowired or constructor injection.

  FIELDS:
    String publicKeyPem  — @Value("${jwt.public-key:}") — RSA PEM from Config Server.
    String jwtSecret     — @Value("${jwt.secret:myVerySecret...}") — HMAC secret,
                           same default as user-service TokenService. Used as fallback.
    PublicKey publicKey  — parsed RSA key, or null if PEM is blank/DUMMY.
    SecretKey hmacKey    — HMAC-SHA256 key built from jwtSecret. Active when publicKey is null.

  METHOD: init() annotated with @PostConstruct
    Two-path initialization:
    Path 1 (RSA): if jwt.public-key is set and does not contain "DUMMY",
      strips PEM headers, base64-decodes, and constructs a PublicKey via
      KeyFactory.getInstance("RSA"). Sets publicKey, leaves hmacKey null.
    Path 2 (HMAC fallback): if PEM is absent or contains DUMMY (current dev config),
      builds an HMAC-SHA256 SecretKey from jwt.secret using Keys.hmacShaKeyFor().
      Pads the key bytes to 32 bytes minimum (JJWT requirement for HS256).
      Sets hmacKey, leaves publicKey null.

    WHY THE HMAC FALLBACK?
      The api-gateway.yml in Config Server ships with a placeholder RSA public key
      ("DUMMY_PUBLIC_KEY_REPLACE_IN_PRODUCTION"). When the gateway detects DUMMY,
      it cannot use RSA. The fallback uses the same HMAC secret that user-service
      TokenService uses to sign tokens, making the gateway able to verify them.
      The key bytes and padding logic in init() must mirror TokenService.getSigningKey()
      exactly, otherwise signature verification fails.

      ORIGINAL BUG: The initial "dev mode" fell back to
        Jwts.parser().build().parseSignedClaims(token)
      without providing any verifying key. JJWT 0.12 throws UnsupportedJwtException
      ("parser was not configured with a verifying key") for any signed token when
      no key is given. This caused every authenticated request to return 401.

  METHOD: validateAndParse(String token)
    Parameters: token - JWT string without "Bearer " prefix
    Returns:    Claims - JJWT Claims object (sub, email, role, jti, exp, iat, iss)
    Throws:     JwtException (covers ExpiredJwtException, MalformedJwtException,
                              SignatureException, UnsupportedJwtException)
    Logic:
      If publicKey != null → verify with RSA:
        Jwts.parser().verifyWith(publicKey).build().parseSignedClaims(token).getPayload()
      Else → verify with HMAC:
        Jwts.parser().verifyWith(hmacKey).build().parseSignedClaims(token).getPayload()
      Both paths verify signature, expiry, and not-before in one call.

PURPOSE:
  Core business logic for user authentication. Handles registration (creates user,
  hashes password, publishes event), login (verifies password, issues tokens),
  token refresh (validates and rotates refresh token), and logout (revokes tokens).

CLASS: AuthService

  ANNOTATION: @Service
    Spring stereotype annotation. Marks this as a service-layer bean. Enables
    Spring to detect it via component scanning and inject it where needed.

  ANNOTATION: @Transactional (class-level)
    All public methods are wrapped in a database transaction by default.
    If a method throws a RuntimeException, the transaction rolls back.
    Individual methods can override with @Transactional(readOnly=true) for
    read-only queries (allows DB to skip write-lock acquisition).

  FIELD: PasswordEncoder passwordEncoder
    BCryptPasswordEncoder bean. encode() applies BCrypt with cost factor 12
    (2^12 = 4096 iterations). Higher cost = slower brute force, but also slower
    login (100ms on modern hardware — acceptable for login, too slow for per-request auth).

  FIELD: UserRepository userRepository
    Spring Data JPA repository interface auto-implemented by Spring Data.

  FIELD: TokenService tokenService
    Handles JWT creation and refresh token Redis operations.

  FIELD: UserEventPublisher eventPublisher
    Publishes domain events to Kafka.

  METHOD: register(RegisterRequest request)
    Parameters:
      request.email    - new user's email address
      request.password - plaintext password (never stored)
    Returns: TokenResponse (access token + refresh token)
    Logic:
      1. Check if email already exists: userRepository.existsByEmail(request.email)
         If true: throw EmailAlreadyExistsException (HTTP 409 Conflict)
      2. Encode password: passwordEncoder.encode(request.password)
         BCrypt automatically includes the salt in the output hash.
         The same password always produces a different hash (salt prevents rainbow tables).
      3. Save User entity to PostgreSQL
      4. Publish UserRegisteredEvent to Kafka "user.registered" topic
      5. Call tokenService.issueTokens(user) and return the result
      The @Transactional annotation ensures that if step 4 or 5 fails, the
      user INSERT is rolled back (atomicity).

  METHOD: login(LoginRequest request)
    Parameters:
      request.email    - existing user's email
      request.password - plaintext password to verify
    Returns: TokenResponse
    Logic:
      1. Load user: userRepository.findByEmail(request.email)
         Throw InvalidCredentialsException (HTTP 401) if not found.
         IMPORTANT: Use the same generic exception for "user not found" and
         "wrong password" to prevent user enumeration attacks.
      2. Verify password: passwordEncoder.matches(request.password, user.getPasswordHash())
         BCrypt.matches() extracts the salt from the stored hash and re-hashes
         the input password. Constant-time comparison prevents timing attacks.
         If false: throw InvalidCredentialsException
      3. Call tokenService.issueTokens(user) and return result

  METHOD: refresh(RefreshRequest request)
    Parameters: request.refreshToken - the opaque refresh token UUID
    Returns: TokenResponse (new access token + new refresh token)
    Logic:
      1. tokenService.validateRefreshToken(request.refreshToken)
         Looks up "refresh:{userId}" in Redis. If key doesn't exist (expired or
         already used), throws InvalidTokenException.
      2. If valid, rotate: delete old key, generate new refresh token, store in Redis
      3. Issue new JWT
      WHY ROTATION? If an attacker steals a refresh token, using it once
      invalidates the old token. The legitimate user's next refresh will fail
      (their token was just used by the attacker), alerting them to a breach.

  METHOD: logout(String userId, String jti)
    Parameters:
      userId - extracted from JWT claims by the gateway (X-User-Id header)
      jti    - JWT ID claim, used as the blacklist key
    Logic:
      1. Delete "refresh:{userId}" from Redis
      2. Add "blacklist:{jti}" to Redis with TTL = remaining JWT lifetime
         Why blacklist instead of just deleting refresh token?
         JWT is stateless — any valid token will be accepted by services
         until it expires (15 minutes). Blacklisting the jti means the gateway's
         JwtAuthFilter can check if the token has been revoked.

CLASS: TokenService

  FIELD: PrivateKey privateKey
    RSA 2048-bit private key loaded from Config Server PEM string.

  FIELD: long REFRESH_TOKEN_EXPIRY_S = 604_800L  (7 days)

    Returns: TokenResponse containing accessToken and refreshToken

             .claim("email", user.getEmail())

    Logic:

      return storedToken != null && storedToken.equals(providedToken)
      NOTE: String comparison here is safe because refresh tokens are full UUIDs (36 chars),
      making timing attacks infeasible at this comparison length.

  to ensure the product write and outbox insert happen atomically.

        - No dirty checking (Hibernate won't scan entities for changes)
        - PostgreSQL can route to read replica
        - Slightly faster due to no flush on commit
    Cache-Aside Implementation:
      1. Optional<String> cached = cacheService.getProduct(id)
         Calls Redis: GET product:{id}
      2. If cached.isPresent():
           return objectMapper.readValue(cached.get(), ProductResponse.class)
           (JSON deserialization from Redis — typically < 1ms)
      3. If cache miss:
           Product product = productRepository.findById(id)
             .orElseThrow(() -> new ProductNotFoundException(id))
           String json = objectMapper.writeValueAsString(product)
           cacheService.setProduct(id, json)  // SET product:{id} {json} EX 600
           return mapper.toResponse(product)

      2. product = productRepository.save(product)
         Hibernate generates UUID and executes INSERT INTO products (...)
      3. Build ProductOutbox entity:
           payload = objectMapper.writeValueAsString(ProductCreatedEvent from product)
           outbox = new ProductOutbox(product.getId(), "product.created", payload)
      4. outboxRepository.save(outbox)
         INSERT INTO product_outbox (..., published=false)
      5. COMMIT (both writes committed together)
      → The OutboxPublisher scheduler will pick this up within 1 second.
    WHY NOT kafkaTemplate.send() HERE?
      If Kafka is temporarily unavailable, the send would fail and we'd need to
      decide: rollback the product? Or save the product with no event published?
      Both options are bad. The outbox defers this decision: the product is always
      saved, and Kafka delivery is retried indefinitely by the scheduler.

      2. Update fields

  METHOD: deleteProduct(UUID id, UUID requesterId)

FILE: product-service/src/main/java/com/ecommerce/product/kafka/OutboxPublisher.java
================================================================================
PURPOSE:
  Scheduled component that polls the product_outbox table every second,
  publishes unpublished rows to Kafka, then marks them as published.
  This is the "relay" half of the Outbox Pattern.

CLASS: OutboxPublisher

  ANNOTATION: @Component
    Spring-managed bean. Receives @Autowired dependencies.

  FIELD: OutboxRepository outboxRepository
  FIELD: KafkaTemplate<String, String> kafkaTemplate
    Spring Kafka's high-level producer API. send() is non-blocking (returns a Future).

  METHOD: publishPendingEvents()
    ANNOTATION: @Scheduled(fixedDelay = 1000)
      Runs every 1000ms (1 second) AFTER the previous execution completes
      (fixedDelay, not fixedRate). fixedDelay is important: if publishing takes
      3 seconds one cycle, the next cycle starts 1 second later (not during).
      This prevents concurrent publishers.

    ANNOTATION: @Transactional
      The SELECT + UPDATE must happen atomically. If we publish to Kafka but
      crash before marking published=true, the next cycle will re-publish.
      Consumers handle idempotency via product ID. The transaction ensures
      we never mark published=true without publishing.

    Logic:
      1. List<ProductOutbox> pending = outboxRepository.findTop100ByPublishedFalseOrderByCreatedAtAsc()
         LIMIT 100 per cycle to avoid memory pressure on large backlogs.
         ORDER BY created_at ensures events are published in creation order.
      2. For each row:
           kafkaTemplate.send(row.getEventType(), row.getAggregateId().toString(), row.getPayload())
             - Topic: row.getEventType() (e.g., "product.created")
             - Key: productId (ensures all events for the same product go to same partition,
               preserving ordering per product)
             - Value: JSON payload string
      3. For each row: row.setPublished(true)
      4. outboxRepository.saveAll(pending)  → batch UPDATE, more efficient than N individual UPDATEs

    WHAT IF KAFKA IS DOWN?
      kafkaTemplate.send() will fail (or timeout). The @Transactional will NOT
      commit the published=true updates. The next scheduled cycle will retry the
      same rows. The product_outbox table accumulates rows during the outage and
      drains automatically when Kafka recovers.

FILE: inventory-service/src/main/java/com/ecommerce/inventory/service/InventoryService.java
================================================================================
PURPOSE:
  Manages stock levels for each SKU. Implements thread-safe reservation using
  JPA optimistic locking. Handles both HTTP-triggered operations (admin) and
  Kafka-triggered operations (order saga).

CLASS: InventoryService

  METHOD: reserveStock(UUID skuId, int quantity, UUID orderId)
    ANNOTATION: @Transactional
    Logic:
      1. Load Inventory by skuId with a SELECT FOR UPDATE (pessimistic read lock)
         or rely on @Version for optimistic locking.
      2. Calculate available: inventory.getQuantity() - inventory.getReservedQuantity()
      3. If available < quantity: throw InsufficientStockException
      4. inventory.setReservedQuantity(inventory.getReservedQuantity() + quantity)
      5. Save: Hibernate executes:
           UPDATE inventory SET reserved_quantity=?, version=? WHERE id=? AND version=?
           The WHERE version=? is the optimistic lock check.
           If version changed since we loaded, this UPDATE matches 0 rows.
           JPA throws OptimisticLockException.
      6. Log InventoryTransaction record for audit trail

    OPTIMISTIC LOCK RETRY (handled in OrderEventConsumer):
      Consumer wraps reserveStock in a retry loop (3 attempts, 100ms backoff).
      Under normal load, contention is rare (most SKUs have separate Inventory rows).
      Under extreme load on a single hot SKU, retries resolve the conflict.

  METHOD: releaseStock(UUID skuId, int quantity, UUID orderId)
    Compensation operation for the Saga.
    Sets reserved_quantity -= quantity.
    Uses same @Transactional + optimistic lock.
    WHY COMPENSATION INSTEAD OF ROLLBACK?
      There is no "distributed rollback" across services. The inventory reservation
      happened in a committed DB transaction. To undo it, we execute a new
      compensating transaction (release).

FILE: inventory-service/src/main/java/com/ecommerce/inventory/kafka/OrderEventConsumer.java
================================================================================
PURPOSE:
  Kafka consumer that listens to the "order.placed" topic. When a new order is
  placed, this consumer attempts to reserve inventory for each item. Publishes
  inventory.reserved or inventory.failed based on the result.

CLASS: OrderEventConsumer

  ANNOTATION: @KafkaListener(topics = "order.placed", groupId = "inventory-service")
    groupId ensures multiple instances of inventory-service share the message load.
    Kafka delivers each partition to exactly one consumer in the group.
    This is how horizontal scaling works for Kafka consumers.

  METHOD: handleOrderPlaced(String payload, Acknowledgment ack)
    Parameters:
      payload - JSON string of the OrderPlacedEvent
      ack     - Manual acknowledgment handle
    ANNOTATION: @KafkaHandler on the method (or @KafkaListener on method directly)

    Logic:
      1. Deserialize payload to OrderPlacedEvent
      2. Try to reserve stock for each item in event.getItems():
           inventoryService.reserveStock(item.getSkuId(), item.getQuantity(), orderId)
      3. If ALL items reserved successfully:
           Publish inventory.reserved event via InventoryEventPublisher
      4. If ANY item fails (InsufficientStockException):
           Release any already-reserved items in this order (partial reservation rollback)
           Publish inventory.failed event
      5. ack.acknowledge()  → commit the Kafka offset ONLY after processing completes

    WHY MANUAL ACK?
      By default, Spring Kafka auto-commits offsets. If auto-committed before
      processing completes, a crash during processing means the message is lost.
      Manual ack commits the offset only after the entire handler succeeds,
      guaranteeing at-least-once processing.

    WHY AT-LEAST-ONCE (not exactly-once)?
      Exactly-once in Kafka requires Kafka transactions (heavy). We achieve the
      same outcome by making the handler idempotent: check if a reservation for
      this orderId already exists before reserving. If it does, skip silently.

    The intercept() method receives the request and a next handler.

  METHOD: intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>>

      product-service performs a SQL ILIKE search on name when q is present;
      returns all ACTIVE products when q is absent.
      Response shape: Spring Page {content, totalElements, totalPages}.
      Dispatches loadProductsSuccess or loadProductsFailure.

    WHY switchMap?
      User types "lapt" (request 1 sent), then types "laptop" (request 2 sent).
      switchMap cancels request 1 (unsubscribes from it) when request 2 arrives.
      Using mergeMap instead would deliver both responses — the user might see
      results for "lapt" appear after results for "laptop".

    WHY debounceTime(300)?
      Prevents an HTTP request on every keystroke. Waits until the user pauses
      typing. 300ms is a well-researched threshold for "deliberate pause in typing".

PURPOSE:

      - An @Input reference changes
      - An async pipe emits
      - An event fires within the component

      debounceTime(300),

      If you subscribe to an observable without unsubscribing, the subscription

      takeUntil(this.destroy$) automatically unsubscribes when the component is destroyed.
      destroy$ is a Subject that emits in ngOnDestroy().

FILE: infrastructure/init-scripts/03-inventory-db.sql
================================================================================
PURPOSE:
  Initializes the inventory-service database schema. Two tables: inventory
  (stock ledger with optimistic locking) and inventory_transaction (audit log).
  pgcrypto extension is enabled for gen_random_uuid().

SQL TABLES:

  TABLE: inventory
    id                UUID PRIMARY KEY    - Surrogate primary key
    sku_id            UUID UNIQUE         - Stock Keeping Unit identifier. UNIQUE because
                                            each SKU has exactly one inventory row.
                                            The SKU is what gets reserved/released.
    product_id        UUID                - Which product this SKU belongs to. NOT a FK
                                            (product is in a different database/service).
    quantity          INT DEFAULT 0       - Total units in stock (reserved + available).
                                            CHECK (quantity >= 0) prevents negative stock.
    reserved_quantity INT DEFAULT 0       - Units reserved by pending orders. Cannot exceed
                                            quantity. CHECK (quantity >= reserved_quantity).
    version           BIGINT DEFAULT 0    - Optimistic lock version. Hibernate increments
                                            this on every UPDATE. The WHERE clause in the
                                            UPDATE checks this value — if it changed since
                                            the row was read, the update touches 0 rows and
                                            JPA throws OptimisticLockException.

    CONSTRAINT chk_quantity  CHECK (quantity >= 0)
    CONSTRAINT chk_reserved  CHECK (reserved_quantity >= 0)
    CONSTRAINT chk_available CHECK (quantity >= reserved_quantity)
      The three CHECK constraints are enforced at the DB level.
      Even if application logic has a bug, the DB rejects invalid states.

  TABLE: inventory_transaction
    id         UUID PRIMARY KEY  - Audit record ID
    sku_id     UUID NOT NULL     - Which SKU this transaction affects (soft FK)
    type       VARCHAR(50)       - Transaction type: RESERVE, RELEASE, RESTOCK, SELL
                                   Provides full audit trail of every stock movement.
    quantity   INT NOT NULL      - Units involved (positive for RESTOCK/SELL,
                                   can be conceptually negative for RESERVE/RELEASE
                                   but always stored as positive absolute value)
    order_id   UUID              - Associated order ID (nullable — RESTOCK has no order)
    created_at TIMESTAMPTZ       - When this stock movement occurred

PURPOSE:
  Initializes the payment-service database schema. Two tables: payments
  (one row per charge attempt) and payment_outbox (Outbox Pattern for Kafka publishing).

SQL TABLES:

  TABLE: payments
    id              UUID PRIMARY KEY      - Payment record ID
    order_id        UUID NOT NULL         - Which order this payment belongs to.
                                            NOT a FK (order DB is separate).
    amount          NUMERIC(12,2)         - Exact decimal amount charged. Never FLOAT.

                                            even if the application check misses a race,
                                            the DB will reject a duplicate INSERT.
    created_at      TIMESTAMPTZ           - Payment attempt timestamp

  INDEX: idx_payments_order_id ON payments(order_id)

  TABLE: payment_outbox
    id           UUID PRIMARY KEY  - Outbox record ID
    aggregate_id UUID              - The payment ID (or orderId) this event relates to
    event_type   VARCHAR(100)      - "payment.completed" or "payment.failed"
    payload      JSONB             - Full event JSON. JSONB enables indexed queries.
    created_at   TIMESTAMPTZ       - When the payment event was generated
    published    BOOLEAN           - FALSE until OutboxPublisher sends to Kafka

  INDEX: idx_payment_outbox_unpublished ON payment_outbox(published, created_at)
         WHERE published = false
    Partial index — same rationale as product_outbox. Only indexes the rows the
    OutboxPublisher needs to read, making the poll query O(log n) on unpublished set.

FILE: service-registry/Dockerfile
  (shared pattern — all 9 Java service Dockerfiles follow this same template)
================================================================================
PURPOSE:
  Multi-stage Docker build for Spring Boot microservices.
  Stage 1 (build): Compiles the Maven project and produces the JAR.
  Stage 2 (runtime): Copies only the JAR into a minimal JRE image.
  This eliminates Maven, the JDK, all source code, and build artifacts from the
  final image — resulting in a much smaller image (~100MB vs ~500MB).

STAGE 1: maven:3.9.6-eclipse-temurin-17 AS build
  COPY pom.xml .
  RUN mvn dependency:go-offline -B
    Downloads all Maven dependencies into the Docker layer cache. This layer is
    cached as long as pom.xml doesn't change. Subsequent builds where only
    source code changes skip the Maven download phase entirely (fast rebuilds).
  COPY src ./src
  RUN mvn package -DskipTests -B
    Compiles and packages the JAR. -DskipTests skips unit tests in Docker build
    (tests run separately in CI pipeline). -B = batch mode (no interactive prompts).

STAGE 2: eclipse-temurin:17-jre-alpine
  eclipse-temurin: Adoptium distribution (reliable, tested, free).
  17-jre: JRE only — no javac, no JDK tools. Smaller attack surface.
  alpine: Minimal Linux base (~5MB). Keeps the image small.
  COPY --from=build /app/target/*.jar app.jar
    Copies ONLY the JAR from the build stage. All Maven cache and source are
    discarded. This is the multi-stage build's key benefit.
  EXPOSE: The service's port (varies per service: 8761, 8888, 8080, etc.)
  ENTRYPOINT ["java", "-jar", "app.jar"]
    Uses exec form (JSON array) not shell form. Exec form is preferred because
    the JVM process becomes PID 1, receiving SIGTERM directly on docker stop.
    Shell form wraps in /bin/sh -c, which doesn't propagate signals correctly.

frontend/Dockerfile DIFFERENCES:
  Stage 1: node:20-alpine — runs npm install + ng build --configuration production
    NOTE: Uses npm install (not npm ci) because no package-lock.json is committed.
    npm ci requires a lockfile; npm install resolves from package.json directly.
  Stage 2: nginx:alpine — serves the compiled static files
  nginx.conf is copied to configure routing and API proxy.

frontend/nginx.conf:
  location /:
    try_files $uri $uri/ /index.html
    Angular uses HTML5 routing (pushState). Without this, refreshing on
    /products/123 returns 404 because nginx looks for a file at that path.
    Falling back to /index.html lets Angular's router handle the path client-side.
  location /api/:
    proxy_pass http://api-gateway:8080
    Any request to /api/* is proxied to the API Gateway container.
    This avoids CORS entirely: the browser talks to localhost (same origin),
    nginx forwards to the gateway behind the scenes.

    proxy_http_version 1.1 + proxy_set_header Connection "":
    CRITICAL for correctness. nginx defaults to HTTP/1.0 for upstream connections.
    Spring Boot sends paginated responses (Page<T>) with Transfer-Encoding: chunked
    because the content-length is not known upfront. HTTP/1.0 has no chunked encoding
    support, so nginx drops trailing chunks, causing ERR_INCOMPLETE_CHUNKED_ENCODING
    (200 OK in the browser but the response body is truncated). Setting HTTP/1.1 and
    clearing the Connection header enables persistent connections and chunked encoding.

    proxy_read_timeout 60s / proxy_connect_timeout 10s:
    Explicit timeouts. Default proxy_read_timeout is 60s but setting it explicitly
    prevents it being inherited from a wider server block that might be shorter.

FILE: service-registry/src/main/resources/application.yml
================================================================================
PURPOSE:
  Configures the Eureka server itself. The key settings prevent the server from
  trying to register with itself (it IS the registry) and fetch its own registry.

KEY SETTINGS:

  eureka.client.register-with-eureka: false
    Tells this instance not to register itself with Eureka. Without this, the server
    would try to register with itself and fail on startup.

  eureka.client.fetch-registry: false
    Tells this instance not to fetch the registry from a peer. Without this, it would
    try to fetch from itself.

  eureka.server.wait-time-in-ms-when-sync-empty: 0
    By default, Eureka waits 5 minutes after startup before serving clients while
    it waits for other registry peers. Since we have only one instance, setting 0
    makes it available immediately.

  eureka.server.use-read-only-response-cache: false
    Eureka has two internal caches: a read-write cache updated on every
    registration/heartbeat, and a read-only cache refreshed from it every 30 seconds.
    Client GET requests hit the read-only cache. With this default, a newly registered
    service is invisible to other clients for up to 30 seconds after it registers.
    In development (and after service-registry restarts), this creates a window where
    all services re-register but the gateway still sees "Registered Applications: 0"
    and returns 503. Setting use-read-only-response-cache: false makes registrations
    visible immediately via the read-write cache.

  eureka.server.response-cache-update-interval-ms: 1000
    How often the read-only cache syncs from the read-write cache (only relevant
    if use-read-only-response-cache is true). Set to 1 s as belt-and-suspenders.

  eureka.server.enable-self-preservation: false
    Self-preservation mode kicks in when Eureka stops receiving heartbeats from
    more than 85% of registered instances (e.g., during a network partition). In
    that state Eureka refuses to evict any registrations even if services are dead,
    to avoid mass de-registration on a network blip. In development this causes
    stale entries to persist in the registry long after a service is killed or
    restarted. Disabling it means dead services are evicted promptly.

  eureka.server.eviction-interval-timer-in-ms: 5000
    How often Eureka runs the eviction task to remove instances that have not sent
    a heartbeat within their lease-expiration window. Default is 60 seconds.
    At 5 seconds, a dead service disappears from the registry within ~20 seconds
    instead of ~90 seconds.

  management.endpoints.web.exposure.include: health,info
    Only health and info endpoints are exposed (no prometheus, no env exposure).
    The registry does not generate business metrics — just needs health for Docker.

    repositories, and is easy to extend (e.g., skip if count > 0).

FILE: product-service/src/main/java/com/ecommerce/product/security/HeaderAuthFilter.java
================================================================================
PURPOSE:
  A servlet filter that translates the trusted request headers injected by the
  API Gateway into a Spring Security Authentication object for the current thread.
  This lets @PreAuthorize expressions on controller methods read the role of the
  caller without each method re-parsing headers manually.

CLASS: HeaderAuthFilter extends OncePerRequestFilter

  EXTENDS: OncePerRequestFilter
    Spring's base class for filters that should execute at most once per request.
    Prevents accidental double-execution in forwarded or included requests.

  METHOD: doFilterInternal(HttpServletRequest, HttpServletResponse, FilterChain)
    Logic:
      1. Read "X-User-Id" header.
         Set by the API Gateway's JwtAuthFilter from the JWT "sub" claim.
      2. Read "X-User-Role" header.
         Set by the API Gateway's JwtAuthFilter from the JWT "role" claim.
         Values: "ADMIN", "USER", or "VENDOR" (uppercase, from the Role enum).
      3. If BOTH headers are present:
           SimpleGrantedAuthority authority = new SimpleGrantedAuthority("ROLE_" + role.toUpperCase())
           UsernamePasswordAuthenticationToken auth =
               new UsernamePasswordAuthenticationToken(userId, null, List.of(authority))
           SecurityContextHolder.getContext().setAuthentication(auth)
         WHY "ROLE_" PREFIX?
           Spring Security's hasRole('ADMIN') check internally looks for the authority
           string "ROLE_ADMIN". If we stored just "ADMIN", hasRole('ADMIN') would fail.
           hasAuthority('ADMIN') would work without the prefix but hasRole() is the
           conventional idiomatic form in Spring Security.
         WHY userId AS PRINCIPAL?
           The principal (first constructor argument) is the name returned by
           authentication.getName(). Storing userId here makes it accessible via
           @AuthenticationPrincipal or SecurityContextHolder if needed later.
      4. chain.doFilter(request, response)
         Always continues the filter chain, even if headers are absent. HTTP-level
         permit-all in SecurityConfig allows unauthenticated requests to pass
         (GET endpoints). @PreAuthorize blocks unauthorized access to write endpoints.

  TRUST MODEL:
    This filter trusts X-User-Id and X-User-Role headers unconditionally. This is
    safe ONLY because:
      a) The API Gateway validated the JWT before forwarding the request.
      b) Services are not directly reachable from the internet (only through the gateway).
    In a zero-trust setup, the product-service would validate the JWT itself using
    the RSA public key, rather than trusting gateway-injected headers.

FILE: inventory-service/src/main/java/com/ecommerce/inventory/entity/InventoryTransaction.java
================================================================================
PURPOSE:
  JPA entity for the inventory_transaction table. Immutable audit log entry.

FIELDS:

  skuId     @Column(nullable=false) — Which SKU was affected
  type      String                  — RESERVE (hold for order), RELEASE (cancel reservation),

PURPOSE:

  already committed to the DB; a Kafka publish failure should be retried (via

FILE: notification-service/src/main/java/com/ecommerce/notification/consumer/UserEventConsumer.java
================================================================================
PURPOSE:
  Kafka consumer for user lifecycle events. Triggers welcome emails for new registrations.

CLASS: UserEventConsumer

  METHOD: onUserRegistered(ConsumerRecord record, Acknowledgment ack)
    ANNOTATION: @KafkaListener(topics="user.registered", groupId="notification-service")

    Logic:
      1. Parse JSON event from record.value() using ObjectMapper.readTree()
         readTree() returns JsonNode — a generic JSON tree. Avoids needing a
         dedicated event POJO for a simple two-field extraction.
      2. Extract "email" field. Falls back to "unknown@example.com" if missing
         (defensive: never crash on malformed events).
      3. Extract "username" field. Falls back to "Customer" if missing.
      4. emailService.sendWelcomeEmail(email, name)
      5. ack.acknowledge() — commits the Kafka offset.

    ERROR HANDLING: ack.acknowledge() is called even on exception.
      WHY? Bad/malformed events should not block the consumer. Without acknowledging,
      the consumer retries the same bad message forever (poison pill).
      The error is logged; in production, redirect to a Dead Letter Topic.

PURPOSE:

FILE: notification-service/src/main/java/com/ecommerce/notification/service/EmailService.java
================================================================================
PURPOSE:

  to stdout instead of failing the notification flow.

CLASS: EmailService

  FIELD: JavaMailSender mailSender
    Spring's mail abstraction. In production, configure spring.mail.host/port/credentials

FILE: frontend/src/app/core/interceptors/error.interceptor.ts
================================================================================
PURPOSE:
  Handles 401 responses by attempting a token refresh before propagating the error.
  Acts as the 401 recovery mechanism — complements the AuthInterceptor which adds tokens.

CLASS: ErrorInterceptor implements HttpInterceptor

  METHOD: intercept(req, next)
    Calls next.handle(req) and pipes catchError to intercept HTTP errors.

    On 401 (Unauthorized):
      1. Gets refresh token from TokenStorageService.
      2. If refresh token exists:
           Calls authService.refreshToken(refreshToken) — Observable<AuthTokens>
           switchMap: on new tokens, stores them and retries the original request
             with the new Authorization header.
           catchError on refresh failure: clears localStorage and navigates to /login.
             The refresh token itself has expired (7-day TTL). User must re-authenticate.
      3. If no refresh token (user was never logged in):
           Clear tokens, navigate to /login.

    On other errors (400, 403, 500, etc.):
      throwError(() => error) — propagates to the component's catchError or error$

  DIFFERENCE FROM auth.interceptor.ts:
    AuthInterceptor ADDS tokens to outgoing requests (proactive).
    ErrorInterceptor REACTS to 401 responses (reactive).
    Both are needed: AuthInterceptor handles normal requests; ErrorInterceptor
    handles the JWT-expired case gracefully.

FILE: frontend/src/app/store/auth/auth.effects.ts

PURPOSE:
  NgRx Effects for auth actions. All HTTP calls and side effects (localStorage,
  navigation) happen here, not in components or reducers.

EFFECTS:

  login$: ofType(login) → switchMap → authService.login() → loginSuccess or loginFailure
    On success: decodes JWT payload (base64url decode of middle segment) to extract
    user info (id, email, role). Stores tokens and user in localStorage via
    TokenStorageService. Dispatches loginSuccess.
    JWT decode: atob(tokens.accessToken.split('.')[1])
      split('.') gives [header, payload, signature].
      atob() decodes base64url to string. JSON.parse() converts to object.
      NOTE: This is client-side decode only (no signature verification).
      The server already validated the signature. Client decodes claims for display.

    { dispatch: false } — this effect does not dispatch a new action.
    Side-effect only: navigates to product catalog after successful login.

    catchError: even if the logout HTTP call fails (expired token, network error),
    we clear localStorage and dispatch logoutSuccess. Logout must always succeed locally.

  logoutSuccess$: navigates to /login.

  register$: ofType(register) → authService.register() → registerSuccess or registerFailure
    catchError extracts the message with:
      err?.error?.message ?? err?.error?.error ?? err?.message ?? 'Registration failed...'
    The backend sends {"error": "Email already registered: ..."} (field name is "error",
    not "message"). Checking err?.error?.error surfaces the real conflict message.
    Same pattern applied to login$ catchError for {"error": "Invalid credentials"}.

DESIGN PATTERN IMPLEMENTATION NOTES — COMMON MISTAKES TO AVOID
================================================================================

OUTBOX PATTERN:
  WRONG: Save entity → commit → call kafkaTemplate.send() (dual-write, no atomicity)
  RIGHT: Save entity + outbox row in SAME transaction, let scheduler publish

OPTIMISTIC LOCKING:
  WRONG: SELECT stock, calculate, UPDATE in separate transactions (race condition)
  RIGHT: @Version field, Hibernate adds AND version=? to UPDATE automatically

SAGA COMPENSATION:
  WRONG: Try to "rollback" other services' database transactions (impossible)
  RIGHT: Execute new compensating operations (release inventory, refund payment)

CQRS:
  WRONG: Query handler replays all events to compute current state on every read
  RIGHT: Maintain a pre-built read model (orders_read_model) updated by projector

JWT SECURITY:
  WRONG: Store user info in JWT, trust all claims without re-validation
  RIGHT: Validate signature + expiry in gateway, use extracted claims for routing only

CACHE INVALIDATION:
  WRONG: Update DB, forget to invalidate cache (stale data for up to TTL duration)
  RIGHT: Invalidate cache in SAME @Transactional method as DB update

EVENT SOURCING:
  WRONG: Store only current state, overwrite on update (loses history)
  RIGHT: Append-only event log; current state = replay of all events

IDEMPOTENCY:
  WRONG: Process every Kafka message assuming it arrives exactly once
  RIGHT: Check if effect already applied (by orderId, eventId) before processing

FILE: order-service/src/main/java/com/ecommerce/order/projection/OrderProjector.java
================================================================================
PURPOSE:
  Implements the Event Sourcing projection: given an orderId, replays all events
  from the event store in sequence and returns the derived current state.

  This is the "rebuild from history" capability. It's not used on every read
  (the orders_read_model table handles that). Instead, it's used for:
    1. Time-travel debugging: replay up to event N to see past state
    2. Rebuilding a corrupted read model from the authoritative event log
    3. Creating new projections (e.g., a "vendor orders" view) without new writes

CLASS: OrderProjector

  ANNOTATION: @Service
  ANNOTATION: @Transactional(readOnly = true) — reads events, does not write

  FIELD: OrderEventRepository eventRepository
  FIELD: OrderReadModelRepository readModelRepository
  FIELD: ObjectMapper objectMapper

  METHOD: project(UUID orderId)
    Parameters: orderId — the order to project
    Returns:    OrderReadModel — the reconstituted state (NOT saved to DB, returned in memory)
    Logic:
      1. Load all events: eventRepository.findByOrderIdOrderBySequenceNoAsc(orderId)
         ORDER BY sequence_no is critical: events must be applied in insertion order.
         Without ordering, replaying events 3-1-2 would give wrong state.
      2. Create empty OrderReadModel with id = orderId
      3. For each event in sequence: call applyEvent(model, event)
      4. Return completed model

  METHOD: applyEvent(OrderReadModel model, OrderEventEntity event) (private)
    Parameters:
      model  — the read model being built (mutable within this method)
      event  — one event from the event store (immutable)
    Logic (switch on eventType string):
      "OrderPlacedEvent":
        Parse payload JSON to extract userId, total, items.
        Set model.userId, model.status = PENDING, model.total, model.itemsJson.
      "InventoryReservedEvent":
        No state change to display fields. Status stays PENDING.
      "OrderConfirmedEvent":
        Set model.status = CONFIRMED.
      "OrderCancelledEvent":
        Set model.status = CANCELLED.
      default:
        Log debug and skip (forward-compatible: new event types don't break replay).

    ERROR HANDLING:
      Wraps JSON parsing in try/catch. If one event fails to parse, logs the error
      and continues with other events. This ensures a single malformed event does
      not prevent the entire order from being projected.

FILE: api-gateway/src/main/java/com/ecommerce/gateway/config/SecurityConfig.java
================================================================================
PURPOSE:
  Configures Spring Security for the reactive gateway. The gateway uses WebFlux
  (not Servlet), so it needs ServerHttpSecurity instead of HttpSecurity.

  The key purpose is to DISABLE Spring Security's default behaviors which would
  interfere with the custom JWT filter chain:
    - Default: Spring Security requires all requests to be authenticated (403 if not)
    - With SecurityConfig: all paths are permitted (auth is handled by JwtAuthFilter)
    - Default: CSRF protection enabled (causes POST failures without CSRF token)
    - With SecurityConfig: CSRF disabled (REST APIs use JWT, not cookies)
    - Default: HTTP Basic login form enabled
    - With SecurityConfig: disabled (no browser login form for an API gateway)

CLASS: SecurityConfig

  ANNOTATION: @Configuration
  ANNOTATION: @EnableWebFluxSecurity
    Enables Spring Security's reactive (WebFlux) security infrastructure.
    Required for SecurityWebFilterChain beans to be picked up.
    Must use @EnableWebFluxSecurity (not @EnableWebSecurity) because the gateway
    runs on WebFlux/Netty, not on Servlet/Tomcat.

  METHOD: securityWebFilterChain(ServerHttpSecurity http)
    Returns: SecurityWebFilterChain — the active security filter chain
    Configuration:
      .csrf(disable)
        Disables CSRF tokens. REST API clients (Angular SPA, mobile) cannot
        send CSRF tokens. Authentication is via JWT in Authorization header.
      .authorizeExchange(exchanges -> exchanges.anyExchange().permitAll())
        Permits all requests at the Spring Security layer. Real auth happens
        in JwtAuthFilter (a custom GlobalFilter). This avoids double-auth.
      .httpBasic(disable)
        Removes the WWW-Authenticate: Basic header from 401 responses.
        Browsers would show a popup login dialog without this — undesirable.
      .formLogin(disable)
        Removes the default /login form. The Angular app has its own login page.

  WHY TWO LAYERS (SecurityConfig + JwtAuthFilter)?
    SecurityConfig manages Spring Security infrastructure setup.
    JwtAuthFilter is the application-level auth logic.
    Separating them makes JwtAuthFilter testable independently of Security config.

FILE: frontend/src/app/app-routing.module.ts

PURPOSE:
  Defines the application's top-level route configuration. Uses lazy loading for
  all feature modules — they are NOT bundled in the main.js but downloaded on demand.

ROUTES:
  { path: '', redirectTo: 'products', pathMatch: 'full' }
    Default route. Navigating to '/' redirects immediately to '/products'.
    pathMatch: 'full' ensures only the exact empty path matches (not prefixes).

  { path: '', loadChildren: AuthModule }
    Loads AuthModule for ANY path not matched by prior routes.
    AuthModule internally declares routes for 'login' and 'register'.
    Using empty string '' as parent means /login and /register work as root paths.

  { path: 'products', loadChildren: CatalogModule }
    Lazy-loads CatalogModule when any /products path is accessed.
    CatalogModule declares:
      /products       → ProductListComponent
      /products/:id   → ProductDetailComponent

  { path: 'cart', loadChildren: CartModule, canActivate: [AuthGuard] }
    AuthGuard runs before loading CartModule. If the user is not logged in,
    they are redirected to /login immediately (the module is never loaded).

  { path: 'orders', loadChildren: OrdersModule, canActivate: [AuthGuard] }
    Same pattern. Order history requires authentication.

  { path: 'admin', loadChildren: AdminModule, canActivate: [AuthGuard, AdminGuard] }
    Two guards run in sequence:
      1. AuthGuard — redirect to /login if not logged in at all.
      2. AdminGuard — redirect to /products if logged in but not ADMIN role.
    Both guards must return true for the route to activate.
    AdminModule lazy-loads only when an admin navigates to /admin.

  { path: '**', redirectTo: 'products' }
    Catch-all for unknown routes. Redirects to product list instead of showing
    a blank page or 404 error.

  LAZY LOADING BENEFIT:
    The Angular build outputs separate JS chunk files per feature module.
    A user browsing the catalog never downloads the cart or orders code.
    This reduces Time-to-Interactive for the first page load.

PURPOSE:
  Singleton module for app-wide infrastructure. Imported ONCE in AppModule.
  Provides HTTP interceptors that wrap every HttpClient request.

  The @Optional @SkipSelf guard pattern in the constructor detects if CoreModule
  is accidentally imported twice (e.g., in a feature module). If imported twice,
  interceptors would be registered twice, doubling every HTTP call's overhead.

CLASS: CoreModule (NgModule)

  IMPORTS: [CommonModule]

  PROVIDERS:

      Adds AuthInterceptor to the HTTP interceptor chain.
      multi: true means "add to the array" (not replace). Without multi, the
      next interceptor registration would overwrite this one.
      Interceptors run in registration order: CorrelationId → Auth → Error → send request.

    { provide: HTTP_INTERCEPTORS, useClass: ErrorInterceptor, multi: true }
      Adds ErrorInterceptor for centralized error handling (401 token refresh, 429 toast, 5xx notification).

  CONSTRUCTOR GUARD:
    constructor(@Optional() @SkipSelf() parent: CoreModule) {
      if (parent) throw new Error('CoreModule already loaded')
    }
    @SkipSelf(): When checking for CoreModule, look in the PARENT injector (not current).
    @Optional(): Do not throw an error if the parent injector doesn't have CoreModule.
    The combination: "check if a parent already provided CoreModule; if so, throw."

FILE: frontend/src/app/store/products/products.effects.ts
================================================================================
PURPOSE:
  NgRx Effects for the products and search state. Effects handle side effects
  (HTTP requests) in response to NgRx actions, then dispatch success or failure actions.
  Also handles admin write operations (create/update/delete) with Router navigation.

CLASS: ProductsEffects
  INJECTED: HttpClient, Router

  EFFECT: loadProducts$
    Triggers on: loadProducts action
    Logic:
      Always calls GET /api/products (product-service catalog endpoint) with params:
        page, size (always)
        q={query} (only when query is non-empty)
      The product endpoint returns a Spring Page: {content, totalElements, totalPages}.
      Maps content to products array and dispatches loadProductsSuccess.
      On any error: dispatches loadProductsFailure with error message.

      WHY NOT /api/search?
        The Elasticsearch pipeline (outbox → Kafka → search-service → ES) has multiple
        silent failure points. ProductEventConsumer swallows all indexing exceptions;
        Kafka delivery can fail silently. Rather than rely on that async chain for basic
        search, the product catalog endpoint now supports a `q` parameter that does a
        SQL ILIKE search directly in PostgreSQL — always consistent and immediately
        available on every startup.

    OPERATOR: switchMap
      Cancels any in-flight HTTP call when a new loadProducts is dispatched.
      Prevents stale results from appearing if the user types quickly.

  EFFECT: loadProduct$
    Triggers on: loadProduct action (single product by ID)
    Calls: GET /api/products/{id}
    Returns individual product with full details.
    On success: dispatches loadProductSuccess.
    On failure: dispatches loadProductFailure.

  EFFECT: createProduct$
    Triggers on: createProduct action
    Calls: POST /api/products (AuthInterceptor attaches JWT)
    On success: dispatches createProductSuccess({ product })
    On failure: dispatches createProductFailure({ error })

  EFFECT: createProductSuccess$  { dispatch: false }
    Side effect only — no new action.
    Calls this.router.navigate(['/admin']) after product is created.
    Navigates admin back to the product list so they can see the new entry.

  EFFECT: updateProduct$
    Triggers on: updateProduct action
    Calls: PUT /api/products/{id}
    On success: dispatches updateProductSuccess({ product })
    On failure: dispatches updateProductFailure({ error })

  EFFECT: updateProductSuccess$  { dispatch: false }
    Calls this.router.navigate(['/admin']) after edit is saved.

  EFFECT: deleteProduct$
    Triggers on: deleteProduct action
    Calls: DELETE /api/products/{id}
    On success: dispatches deleteProductSuccess({ id })
    On failure: dispatches deleteProductFailure({ error })
    No navigation — admin stays on the product list after deletion.

  WHY exhaustMap for write effects?
    Write operations must not be cancelled if a second action arrives.
    switchMap would cancel an in-flight POST/PUT/DELETE — corrupting data.
    exhaustMap ignores new actions while one is in flight (natural rate-limit).

FILE: frontend/src/app/store/cart/cart.reducer.ts
================================================================================
PURPOSE:
  Pure reducer function for the cart state slice. Also exports the localStorage
  meta-reducer for cart persistence.

INTERFACE: CartState
  items: CartItem[]   — Current cart contents
  total: number       — Computed sum of (unitPrice × quantity) for all items
  ordering: boolean   — True while waiting for POST /api/orders response
  error: string | null — Error message from failed order placement

HELPER: calculateTotal(items: CartItem[]): number
  Recomputes the total after every cart mutation. Private to this file.
  Uses Array.reduce() over items: sum += item.unitPrice * item.quantity.
  Stored pre-computed in state so components can select it without re-computing.

REDUCER: cartReducer
  addToCart:
    Find if item with matching skuId exists.
    Existing: map to new array incrementing only that item's quantity.
    New: spread existing items and append new item.
    Both cases: recalculate total.

  removeFromCart:
    Filter out item with matching skuId. Recalculate total.

  updateCartItem:
    Map items to new array replacing only the matching skuId's quantity.
    Recalculate total.

  clearCart:
    Return initialCartState (items=[], total=0, ordering=false, error=null).

  placeOrder:
    Set ordering=true, error=null. UI shows spinner, disables button.

  placeOrderSuccess:
    Set ordering=false. CartEffects separately dispatches clearCart.

  placeOrderFailure:
    Set ordering=false, error=action.error. UI shows error message.

FUNCTION: localStorageCartMetaReducer(reducer): ActionReducer
  Wraps the entire store's reducer (receives full AppState, not just CartState).
  On @ngrx/store/init action: check localStorage['cart']; if present, inject
    it into the initial state so the cart is pre-populated before any renders.
  On every action: serialize nextState.cart to localStorage['cart'].
  Try/catch around localStorage calls: storage may be unavailable (private browsing).

PURPOSE:

      placeOrderSuccess should clear the cart, but placeOrderSuccess is handled
      by the cart reducer (sets ordering=false). If the reducer for placeOrderSuccess
      also cleared items, it would happen in one reducer call. Alternatively,
      dispatching clearCart as a separate action makes the intent explicit in
      the action log (better DevTools debugging).

      categoryId:  [optional]
    isEditMode: boolean — true if route has :id param
    submitting$: Observable<boolean> — disables Save button while HTTP is in flight
    error$: Observable<string | null> — shows error message on failure

  METHOD: ngOnInit()
    this.productId = this.route.snapshot.paramMap.get('id')
    If productId exists (edit mode):
      Dispatch loadProduct({ id: productId }) to fetch current values.
      Subscribe to selectSelectedProduct; when product arrives, call form.patchValue({...}).
    If no productId (create mode): form starts blank.

  METHOD: submit()
    Guard: if form.invalid, return early.
    If edit mode: dispatch updateProduct({ id: productId, product: form.value }).
    If create mode: dispatch createProduct({ product: form.value }).
    Effect handles navigation to /admin on success.

  METHOD: ngOnDestroy()
    Dispatches clearSelectedProduct() to reset selectedProduct in store.
    Prevents stale product data from pre-filling the form on the next edit navigation.

TEMPLATE (product-form.component.html):
  <form [formGroup]="form" (ngSubmit)="submit()">
    Input fields for name, description, price, categoryId.
    <button type="submit" [disabled]="submitting$ | async">
      {{ (submitting$ | async) ? 'Saving…' : (isEditMode ? 'Update Product' : 'Create Product') }}
    </button>
    <a routerLink="/admin">Cancel</a>
    <div *ngIf="error$ | async as err" class="alert-error">{{ err }}</div>

MODULE HIERARCHY:
  AppModule (root, eager)
    ├── CoreModule (singleton infrastructure)
    │     AuthInterceptor, ErrorInterceptor
    ├── SharedModule (NOT imported here — only in feature modules)
    ├── NgRx StoreModule.forRoot()
    │     auth, products, cart reducers + meta-reducers
    ├── NgRx EffectsModule.forRoot()
    │     AuthEffects, ProductsEffects, CartEffects
    └── AppRoutingModule → lazy-loads:
          AuthModule     (/login, /register)
          CatalogModule  (/products, /products/:id)
          CartModule     (/cart)
          OrdersModule   (/orders)
          AdminModule    (/admin, /admin/products/new, /admin/products/:id/edit)

STATE FLOW:
  User action → Component dispatches Action → Effect (HTTP call) → Success/Failure Action
             → Reducer updates state → Selector emits new value → Component re-renders

  Example (Add to Cart):
    ProductListComponent.addToCart(product)
      → dispatch addToCart({ item })
      → cartReducer merges item into state.items and recomputes total
      → localStorageCartMetaReducer serializes state.cart to localStorage
      → AppComponent.cartCount$ emits new count → navbar badge updates
      → CartComponent.items$ emits new list (if cart page is open)

  Example (Search):
    SearchBarComponent emits searchQuery after 400ms debounce
      → ProductListComponent.onSearch(query) dispatches loadProducts({ query, page: 0 })
      → ProductsEffects.loadProducts$ (switchMap cancels any prior request)
        → GET /api/products?q={query}&page=0&size=12
        → product-service: SQL ILIKE search on product name in PostgreSQL
      → dispatch loadProductsSuccess({ products, totalElements, totalPages })
      → productsReducer sets state.products, state.totalElements, state.totalPages
      → ProductListComponent.products$ emits → *ngFor re-renders grid

  Example (Order Placement Saga):
    CartComponent.checkout() dispatches placeOrder()
      → cartReducer sets ordering=true → button shows spinner
      → CartEffects.placeOrder$ reads selectCartItems via withLatestFrom
        → POST /api/orders { items: [...] }
        → API Gateway: validates JWT, injects X-User-Id
        → order-service: PlaceOrderCommandHandler creates event, saga, read model
        → Publishes order.placed to Kafka
        → Returns 202 { orderId, status: "PENDING" }
      → dispatch placeOrderSuccess({ orderId })
        → CartEffects.placeOrderSuccess$: dispatch clearCart(), navigate /orders
        → localStorageCartMetaReducer saves empty cart to localStorage
        → OrderHistoryComponent loads and shows PENDING order

  Example (Admin Create Product):
    Admin navigates to /admin/products/new
      → AdminGuard checks role === 'ADMIN' (from localStorage) → allow
      → ProductFormComponent renders blank form
    Admin fills name/price, clicks Save
      → ProductFormComponent.submit() dispatches createProduct({ product: form.value })
      → productsReducer: { submitting: true } → Save button disables
      → ProductsEffects.createProduct$:
          POST /api/products with JWT (AuthInterceptor attaches Bearer token)
          → API Gateway validates JWT, injects X-User-Role: ADMIN
          → HeaderAuthFilter builds SecurityContext with ROLE_ADMIN
          → ProductController @PreAuthorize("hasRole('ADMIN')") passes
          → Product saved, returns 201 with created entity
      → dispatch createProductSuccess({ product })
      → productsReducer: { submitting: false, products: [new, ...existing] }
      → ProductsEffects.createProductSuccess$: router.navigate(['/admin'])
      → AdminProductListComponent shows new product at top of table
